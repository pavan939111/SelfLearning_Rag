Overall System Architecture
One unified self-healing RAG system operating in two parallel loops — a real-time hot path serving user queries and a continuous cold path healing the corpus. Eight agents. Eight MCP servers. Four cloud databases. All free tier.
System Overview — Two Parallel Loops
Hot path and cold path communicate only through shared database state
⚡ HOT PATH — Real Time (Under 8 seconds P95)
1User query arrives → Redis cache check
2Agent 1 — Query classification + strategy selection
3Agent 1 — Agentic retrieval (5 strategies, iterative eval)
4Agent 1 — Answer generation (Gemini 2.0 Flash)

9Class A/B → Agent 4B queued → Honest confidence flag returned

Agent 1 — Agentic Retrieval and Generation
Five retrieval strategies with iterative evaluation and replanning loop

Query Classification

→
simple_factual
Strategy: single_shot_hybrid — one pass, fast
→
multi_hop
Strategy: iterative_decomposed — decompose → retrieve per sub-Q → accumulate
→
comparative
Strategy: parallel_comparative — extract entities → parallel branches → merge
→
temporal
Strategy: freshness_prioritized — mandatory freshness check → live fetch if stale
→
exploratory
Strategy: broad_then_deep — broad 30 → identify subtopics → deep per subtopic

Retrieval Pipeline (all strategies)

1Dense retrieval top-20 via Qdrant (S-PubMedBert embeddings)
2Sparse BM25 retrieval top-20 via Qdrant sparse vectors
3RRF fusion → unified top-20 ranking
4Metadata filter: remove contradiction_flag=True AND freshness < 0.4
5MMR reranking: top-5 most relevant AND diverse chunks
6Sufficiency evaluation: relevance + freshness + completeness check
7aSUFFICIENT → Generation with Gemini 2.0 Flash
7bINSUFFICIENT + iterations left → replan with different angle
7cINSUFFICIENT + max iterations → live fetch fallback
Agent 2 — Self Evaluation
Five checks run in parallel async — total time ~1.5 seconds
①
Retrieval Relevance
LLM per-chunk relevance classification. YES/NO per chunk. Gemini Flash.
FAIL if 3+ of 5 chunks irrelevant → RETRIEVAL FAILURE
②
Grounding Check
Claim-level support verification. Each answer claim checked against retrieved chunks. Gemini Flash.
FAIL if 20%+ claims unsupported → GROUNDING FAILURE
③
Freshness Check
Metadata analysis. Contradiction flags from Agent 5A. Topic velocity from Agent 6. Age score calculation. Pure math — no LLM.
FAIL if freshness < 0.4 → DRIFT → Agent 4A live fetch
④
Calibration Check
Historical calibration curve lookup from Supabase. Compare expressed vs historical accuracy for this topic+query type. DB lookup — no LLM.
ADJUST: override expressed confidence → calibrated score. Continue.
⑤
Compositional Coherence
Cross-document reasoning check. Only fires when 3+ papers retrieved. Gemini 1.5 Pro for long context. Detects contradictions between retrieved papers.
FAIL if INCOHERENT or CONTRADICTION PRESENT → COMPOSITIONAL FAILURE
Output: Structured evaluation object → failure_types list + failure_class (A/B/C) + calibrated_confidence
Routing: All pass → Response Assembly. Any fail → Agent 3 Root Cause Classification
Agent 3 — Root Cause Classifier
Five sequential diagnostic tests — each confirms or eliminates a root cause hypothesis
T1
Existence Test
Generate 8 alternative query phrasings. Run all against corpus. 6+ failing → COVERAGE FAILURE (Class B — Knowledge Problem). Information does not exist in corpus.
T2
Chunking Boundary Test
Check adjacent chunks of retrieved documents. Relevant info in neighboring chunks (similarity >0.7) → CHUNKING FAILURE (Class A — Data Problem). Information split across boundaries.
T3
Embedding Space Test
Run BM25 on same docs retrieved by vector search. High BM25 but low vector score (gap >0.3) → EMBEDDING FAILURE (Class A). Semantic gap in embedding space.
T4
Query Processing Test
Run original unexpanded query. Compare results. Strategy mismatch check. Improvement without expansion → QUERY FAILURE (Class C — Query Problem).
T5
Metadata Filter Test
Run retrieval without metadata filters. Compare results. Improvement → filters incorrectly removed relevant chunks → METADATA FAILURE (Class A).
Class C → Agent 4A
Query problems. Instant reversible fixes. Threshold: 65% confidence minimum.
Class A/B → Agent 4B
Data or Knowledge problems. Permanent corpus repairs. Threshold: 85% confidence minimum.
Below 65% → Escalate
Admin notified with full evidence chain. Log failure. Monitor for pattern accumulation by Agent 6.
Agent 4A — Instant Repair Agent
Hot path only — all repairs complete within live query cycle — nothing persists to corpus
1
Query Rewriting — for low relevance retrieval / query processing failure
Rewrite query with different vocabulary, structure, or specificity using Gemini Flash. Re-retrieve. Regenerate. Under 3 seconds.
2
Synonym Expansion — for embedding space failure / terminology gap
Expand query with UMLS medical synonyms and abbreviation expansions. heart attack → myocardial infarction. NSCLC → non-small cell lung cancer. Milliseconds.
3
Retrieval Parameter Adjustment — for too few relevant chunks
Increase top-k from 5 to 10. Lower similarity threshold. Re-retrieve with broader net. Milliseconds.
4
Live Fetch Fallback — for knowledge drift / freshness failure (score <0.4)
Parallel calls to PubMed, Semantic Scholar, bioRxiv via MCP servers. Generate from live papers with timestamps. Queue fetched papers for Agent 5A via Redis. 3-5 seconds.
5
Confidence Flagging — when no instant fix possible
Return best available answer with honest calibrated confidence score and explicit uncertainty flags. Milliseconds.
Constraints: Never touches Qdrant, Neo4j, or Supabase corpus data. Never modifies any persistent state. Maximum 2 retry attempts before falling back to confidence flagging.
Feeds: Every successful repair pattern logged to Agent 6. Every live fetch paper queued to Agent 5A for verification.
Agent 4B — Background Repair Agent
Cold path only — Celery workers — always staging first, never touches production directly

Repair Types

→
Re-chunking — chunking boundary failure
Try alternative strategies: sentence, paragraph, semantic unit, sliding window. Evaluate on failing queries. Select best. Auto-approved if <100 docs.
→
Synonym Augmented Indexing — embedding failure
Build synonym pairs for failing terminology. Augment Qdrant index with variants. Always auto-approved.
→
Metadata Correction — metadata filter failure
Correct contradiction flags or freshness scores on affected chunks. Update payload in Qdrant. Always auto-approved.
→
Large Cluster Restructure — high-severity contradiction
Affects 100+ documents. Requires admin approval. Phased promotion with extended validation.

Execution Flow (all repair types)

1Receive repair job from Redis queue (high/medium/low priority)
2Read root cause diagnosis + affected document IDs from Supabase
3Execute repair on staging Qdrant collection
4Run 3 staging validation queries — all must pass
5Admin approval required? → INTERRUPT → wait
6Promote staging → production (atomic swap)
7Log repair outcome to Supabase with before/after scores
8Notify Agent 2 (recalibrate) + Agent 6 (log for learning)
Agent 5A — Relevance Verification Agent
Four sequential checks — each reject terminates immediately — never inserts anything
C1
Domain Filter — cheap, fast — eliminates 95%+ of papers instantly
Compare paper embedding against corpus cluster centroids. Similarity below threshold → REJECT immediately. Gemini Flash for classification. Milliseconds.
C2
Corpus Relationship Test — checks 4 relationship types in parallel
① Contradiction (claim comparison) ② Citation network (Semantic Scholar) ③ Coverage gap match (Agent 6 gap map) ④ High-traffic topic match (query logs). None found → REJECT.
C3
Evidence Quality Assessment — assigns evidence level, flags weak evidence
meta-analysis > RCT > cohort > case series > case report > preprint. Low evidence contradicting high evidence corpus → flag for admin review, continue. Gemini Flash.
C4
Contradiction Confidence Check — three-step deep analysis
Only fires if C2 found contradiction relationship. Claim extraction (ScispaCy+Gemini) → candidate comparison (vector similarity + deep analysis) → context reconciliation (population, temporal, outcome, study design). Confidence assigned.
Above 0.85 → HIGH Priority
Direct contradiction confirmed. Ingest within 6 hours. Admin notification.
0.60-0.85 → MEDIUM Priority
Probable contradiction. Ingest within 24 hours. Admin dashboard notification.
Below 0.60 → LOW / REJECT
If no other reason qualifies — reject on contradiction grounds. Log and monitor.
Agent 5B — Selective Ingestion Agent
Execution only — follows Agent 5A's ingestion instructions precisely — staging before production always
1
Document Fetch
Full text from PMC (open access) via PubMed MCP. Abstract fallback if full text unavailable. Preprint content from bioRxiv/medRxiv MCP.
2
Structure Detection
Detect document type from content analysis. IMRAD → structure-aware. Review → embedding similarity. Abstract only → proposition direct. OCR noisy → sliding window fallback.
3
Hierarchical Chunking (4 levels)
L1: Document embedding (abstract+conclusion). L2: Section chunks (IMRAD). L3A: Semantic chunks within sections (embedding similarity). L3B: Proposition extraction (ScispaCy+Gemini Flash).
4
Complete Metadata Assignment
freshness_score=1.0, topic_cluster assigned, contradiction_flag=False, evidence_level assigned, ingestion provenance recorded. Affected existing chunks: contradiction_flag set to True, freshness lowered.
5
Staging Validation (3 queries must pass)
① Trigger query: does new paper surface correctly? ② Known good queries: does existing retrieval maintain quality? ③ Adversarial probe: no unexpected degradation. ALL must pass.
6
Admin Approval (if large change)
100+ documents affected OR 20%+ corpus change → INTERRUPT NODE 2. Wait for admin. Otherwise auto-promote.
7
Production Promotion + Downstream Notify
Qdrant staging → production. Neo4j graph updated. Supabase ingestion log. Notify Agent 2 + Agent 3 + Agent 6.
Agent 6 — Longitudinal Learning Agent
Observes all agents — detects patterns across time — never acts directly on corpus

Four Responsibilities

R1
Pattern Detection
Topic patterns (which clusters fail most), failure type trends, query type patterns, temporal anomalies (Monday spike → pipeline issue), repair effectiveness patterns (same doc re-chunked 3x → wrong strategy).
R2
Calibration Maintenance
Maintains calibration curves per topic cluster per query type. Detects calibration drift. Updates Agent 2's calibration thresholds. Tracks ECE (expected calibration error) — target below 0.08.
R3
Coverage Gap Intelligence
Aggregates retrieval failures → structured gap map. Prioritizes by query frequency × severity. This map is what Agent 5A Check 2 reads. Updates as ingestions fill gaps.
R4
Strategy Improvement Signals
Measures which strategies/thresholds work in practice. Surfaces recommendations with evidence to admin dashboard. Human approves changes — Agent 6 never auto-modifies strategies.

Information Flow

IN
Reads from all agents via Supabase
Failure logs, repair outcomes, ingestion events, query traces, calibration history, strategy effectiveness records.
OUT
Outputs calibration curves → Agent 2
Updated thresholds for calibration check. Agent 2 reads from Supabase calibration_history table.
OUT
Outputs coverage gap map → Agent 5A
Prioritized gap list. Agent 5A Check 2 reads from Supabase coverage_gap_map table.
OUT
Outputs insights → Admin Dashboard
Pattern descriptions with evidence chains. Recommended actions. Human reviews and approves or dismisses. 7+ day average lead time before user-visible degradation.
Key constraint: Agent 6 NEVER acts directly on the corpus or modifies agent thresholds autonomously. It observes and recommends. Humans approve changes. This prevents feedback loops where Agent 6's own actions create the patterns it responds to.

MCP Architecture
Eight Model Context Protocol servers — four for external data sources, four for internal databases. Agents reason over a tool registry rather than executing hardcoded function calls.
Eight MCP Servers — Complete Reference
All deployed on Railway internal network — agents connect via HTTP — external APIs never touched by agents directly
EXTERNAL DATA SOURCES
PubMed MCP
search papers, fetch abstract by PMID, fetch full text by PMC ID, fetch citing papers
Used by: Agent 1 (live fetch), Agent 5A (monitoring), Phase 2 seeding
Semantic Scholar MCP
get paper details with citations, search papers, get recommendations, citation velocity
Used by: Agent 5A (verification), Agent 6 (citation tracking)
bioRxiv / medRxiv MCP
fetch new papers by date range, fetch by DOI, search preprints
Used by: Agent 5A (daily monitoring loop)
arXiv MCP
search papers by query and category, fetch paper details
Used by: Agent 5A (CS biology preprints)
INTERNAL DATABASES
Qdrant MCP
hybrid search with filters, insert chunks with payload, update payload, delete, create collections, staging swap
Used by: Agent 1 (retrieval), Agent 4B (repairs), Agent 5B (ingestion)
Supabase MCP
log failure, create approval request, update approval status, read coverage gap map, read calibration, write query trace, read benchmark pairs
Used by: all agents for operational data
Neo4j MCP
create paper node, create relationship, get citation chain, get contradictions, update freshness, query temporal graph
Used by: Agent 5B, Agent 5A, Agent 1 (graph-aware retrieval)
Redis MCP
get cached embedding, set cache with TTL, enqueue job with priority, get queue depth per level
Used by: Agent 1 (cache), all agents (job queuing)
Circuit Breaker Pattern — External MCP Servers
Prevents hammering failing external APIs — internal MCP servers use retry-only
